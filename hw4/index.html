Overview

In this homework, we built a real-time cloth simulation using a mass-spring system, implementing Verlet integration for physics , collision handling with spheres and planes, and self-collision detection via spatial hashing. We also implemented a full shader pipeline GLSL covering Blinn-Phong shading, texture mapping, bump and displacement mapping, and environment reflections. It was interesting to see how bump and displacement mapping can produce similar visual effects from the same height map but through different mechanisms.

Part 1: Masses and springs

In this part a grid of masses and springs were implemented in which we were able to visualize a cloth wireframe in a simulation.

Without any shearing constraints, the wireframe will only show horizontal and vertical lines forming a clean lattice since the diagonal components are missing. With only shearing constraints, we will only see diagonal lines crossing in an X patteron over each cell with no vertical or horizontal edges. With all constraints, both vertical, horizontal, and diagonal components are visible and shows a dense mesh which would produce the most realistic behaving cloth.

Part 2: Simulation via numerical integration

In this part, we implement the numerical integration needed to simulate the cloth's motion over time. Each timestep, we compute the total forces acting on every point mass, combining external forces like gravirty with spring forces from Hooke's law. We then use Verlet integration to update each point mass's position and add a deformation constraint to prevent springs from stretching more than 10% from their rest length.

When adjusting the parameters for the spring constant \(ks\), a low constant makes the cloth very loose and causes it sag significantly more. With a higher constant, the cloth becomes very stiff and is harder to "deform" with minimal sagging.

When adjusting the parameters for density, a lower density causes gravity to have a lesser effect on the cloth since the cloth is light. For a higher density, the cloth is more affected by gravity and sags significantly more.

When adjusting the damping, a lower damping causes the cloth to swing and oscillate wildly for a long time before settling. With higher damping, the cloth moves a lot slower and is more sluggish, settling almost immediately.

Part 3: Handling collisions with other objects

For sphere collisions, we check if a point mass has entered the sphere by comparing its distance from the sphere's origin to the radius . First, we find the tangent point by projecting outward from the origin through the point mass's current position to the sphere surface, then compute a vector from last_position to that tangent point.

For plane collisions, we detect crossing by checking whether the dot products of position - point and last_position - point with the plane normal have opposite signs. If they do, the point crossed the plane. We project the current position onto the plane to find the tangent point, then push it slightly above the surface on the side the point came from.

ks comparison on scene/sphere.json

With ks = 5000 (default) it shows a balanced drape and conforms reasonably with sphere. With ks = 500, the cloth conforms more with the shape of the sphere and has many folds hanging low around it. With ks = 50000, the cloth is more stiff and barely conforms to the shape of the sphere.

Part 4: Handling self-collisions

Self-collision is handled using spatial hashing for efficiency. Each timestep, build_spatial_map hashses every point mass's position into a 3D box cell and groups them in a hash table. hash_position partitions space into boxes of size w*h*t (proportional to the cloth dimensions) and maps each box to a unique float. Then self_collide looks up only the point masses sharing the same box cell as candidates, and for any pair closer than 2*thickness, computes a correction vector to push them apart. The final correction applied is the average of all pairwise corrections, scaled down by simulation_steps to maintain a smooth motion.

Varying density

For low density (1g/cm^2), the cloth is light and falls slowly. It folds gently with large smooth waves and doesnt bunch up as aggressively. For high density (50g/cm^2), the cloth is heavy and falls fast, with many crumples and self collisions are frequent as well as bunching up significantly more.

Varying ks

For low ks (500), the cloth is very flexible and folds into many small wrinkles. It collapses easily on itself. For high ks (50000), the cloth is stiff and attempts to resist folding. It has smoother folds as a result, with more defined creases.

Part 5: Shaders

A shader program runs directly on the GPU and determines ow geoemtry and pixels are rendered. The vertex shader processes each vertex individually, which transforms positions from model space to screen space and passing data to the next stage. The fragment shader runs once per pixel fragment, receiving interpolated values from the vertex shader and computing a final output color. This forms the pipeline from vertex shader to fragment shader.

Blinn-Phong Shader

Blinn-Phong approximates how light interacts with a surface using three components. The ambiet term is a constant low light that prevents surfaces from going completely black. The diffuse term dfepends on the angle between the surface normal and the light direction, so surfaces facing the light appear brighter. The specular term creates shiny highlights by measuring how close the surface normal is to the halfway vector between the light and the viewer.

Bump vs Displacement

Bump mapping only perturbs the surface normals used in lighting calculations so the actually geometry actually stays flat. It is very cheap to calculate and looks convincing in most angles, but the object remains smooth. Displacement mapping actually moves vertices along their normals based on the hight map, producing real geometric detail. This makes silhouettes and self-shadowing look correct, but requires a dense mesh to look good. With the parameters -o 16 -a 16, the displacement looks blocky since it models a coarse mesh, whilst -o 128 -a 128 captures the texture detail more. Bump mapping would look similar at both resolutions since it doesn't depend on the mesh.
