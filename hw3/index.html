Overview

In this assignment, I implemented a path tracer that renders realistic images using Monte Carlo sampling. I added direct lighting with both hemisphere and importance sampling, then extended it to global illumination by recursively bouncing light. I also implemented a Bounding Volume Hierarchy in order to speed up these renders by a significant amount. Overall this assignment helped me understand how light is simulated in rendering. I found it interesting how randomness can still produce accurate results, and how techniques like importance sampling and adaptive sampling can reduce noise and improve efficiency.

Part 1: Ray Generation and Scene Intersection

The first step in the rendering pipeline is generating a ray for each pixel on the screen. For each sample, a random offset is generated inside the pixel using a sampler. The pixel coordinate is then converted into normalized image coordinates that are in the range \((0,1)\). These normalized coordinates are passed into Camera::generate_ray().

Inside Camera::generate_ray, the 2D image is mapped onto a virtual sensor plane located at \(z = -1 \) in camera space. A ray is then constructed from the camera origin through this point on the sensor plane. The ray is then transformed into world space using the camera to world (c2w) matrix which is then passed into the scene where intersection tests are performed.

For triangles, I used the Möller-Trumbore algorithm. The idea is to express the triangle using two edge vectors: e1 = p2 - p1 & e2 = p3 - p1. Then set the ray equation equal to those edges: \(o + td = p_1 + ue_1 + ve_2\) and solve for \(u, v ≤ 1\) & \(u + v ≤ 1\)

Similar idea applies to spheres, but substitute the ray equation into the sphere equation. \(||o + td - c||^2 = R^2\) becomes \(at^2 + bt + c = 0\), where a, b, and c are the terms after expanding the first equation. This gives us two possible values of t corresponding to where the ray enters and exits the sphere. The smallest value of t is chosen as it corresponds to where the ray first hits the sphere.

Part 2: Bounding Volume Hierarchy

In this homework, BVH is constructed recursively. For each node, I compute a bounding box that encloses all the primitives of the current scene. If the number of primitives are small, I stop splitting and store them in a leaf node. Otherwise, I choose a splitting axis based on the largest extent of the centroid bounding box (cbbox). By doing this, it divides the geometry along the direction where it is most spread out. I then partition the primitives using the midpoint along that axis and recursively build the left and right child nodes.

The speed up between pre-BVH implementation and BVH is significant. When attempting to render meshedit/cow.dae, it took approximately 30 seconds on my system. After full BVH implementation, it only took .149 seconds. As significant as it is, this improvement is expected as BVH reduces the number of intersection tests per ray. Instead of checking every triangle, the algorithm first tests against bounding boxes, which quickly discards large regions of space that the ray does not intersect.

Part 3: Direct Illumination

There are two approaches implemented used to estimate direct lighting. One of them is uniform hemisphere sampling and the other is importance sampling

In uniform hemipshere sampling, I sample directions uniformly over the hemisphere around the hit point. For each sampled direction, a ray is cast and checked whether it intersects a light source. If it does, then we calculate it contribution using the rendering equation. This is simpler to implement, but the tradeoff is that it is inefficient. Since directions are sampled randomly, most rays do not hit a light source. As a result, the estimator has high variance and the result image is noisy unless large number of samples are used.

In importance sampling, it directly samples directions toward light sources. For each light, I use sample_L to generate a direction from the hit point to the light and compute the incoming radiance. A shadow ray is then cast to check whether the light is visible. If the path is not blocked, the contirbution is added using BDSF, cosine, and pdf. The result is a much more efficient sampling because samples are concentrated towards directions that actually contribute light. As a result, fewer samples are wasted and the resulting images have less noise.

The difference between hemisphere sampling and importance sampling is that hemisphere sampling produces significantly nosier results compared to importance sampling. This is because hemisphere sampling does not take into account where the light sources are, so many sampled directions do not actually contribute light. In contrast, importance sampling focuses directly on the lights, making each sample more useful. This in turn makes importance sampling look better and also run faster than hemisphere sampling.

Part 4: Global Illumination

At each intersection, the algorithm computes direct lighting using one_bounce_radiance. Then it samples a new incoming direction from the BSDF and traces a new ray in that direction. The radiance returned from this new intersection is recursively computed and weighted by the BRDF, cosine term, and the sampling probability (0.7). This process continues until maximum depth is reached, or the path is terminated through Russian Roulette.

Part 5: Adaptive Sampling

Adaptive sampling improves rendering efficiency by avoiding unnecessary samples on pixels that have already converged. Instead of using a fix number of samples per pixel, the renderer dynamically decides when to stop sampling based on how stable the pixel's color estimate is. Within my implementation, we keep track of two values. \(s_1\) keeps track of the sum of sample illuminance and \(s_2\) tracks thhe sum of squared illuminance. I then use these values to compute the mean, the variance, and the confidence interval (\(I\)) with 95% confidence. If \(I\) is less than the maxTolerance * mean, then the pixel is converged and sampling stops.

In the sampling rate images, regions with high detail such as object edges, shadow boundaries, and indirect lighting areas required significantly more samples, appearing brighter in the visualization. In contrast flat and uniformly lit areas converged quickly and used fewer samples. Despite using adaptive sampling, the final images appear nearly noise-free and visually similar to high-sample renders without adaptive sampling. However, it is to take note that adaptive sampling is significantly more efficient than sampling more as it focuses computation where it is most needed.
